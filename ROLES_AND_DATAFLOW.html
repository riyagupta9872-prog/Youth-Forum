How roles, permissions, and data move through the system across all tabs

1 · Three Roles

Super Admin

role: "superAdmin"

Super Admin
  • All tabs, all teams
  • Can mark "Not Interested"
  • Manages user logins & roles
  • Can clear session data
  • Sees every team's calling & reports
First user who signs up automatically becomes superAdmin.

Team Coordinator

role: "teamAdmin"

Lalita - Coordinator
  • All tabs, own team only
  • Manages devotees in their team
  • Submits calling for their callers
  • Views reports scoped to team
  • Cannot mark Not Interested
Team filter is locked and disabled in the UI.

Calling Sevak

role: "serviceDevotee"

Lalita - Facilitator
  • All tabs, own team only
  • Calls their assigned devotees
  • Submits calling status
  • Marks attendance in their team
  • Position label set by superAdmin
Header pill shows TeamName - position where position is a free-text field (e.g. "Facilitator") set in Admin Panel.

4 · AppState — Key Fields

userRole
Controls tab visibility, data scoping, and button rendering. Checked in almost every function.
userTeam
Team name for non-superAdmins. Applied as a hard filter on all Firestore queries for teamAdmin and serviceDevotee.
userPosition
Free-text position label (e.g. "Facilitator") set by superAdmin. Shown in header pill as "TeamName - Position".
currentSessionId
Active Sunday session being used for attendance marking. Set by initSession() on load.
currentDevoteeId
Devotee whose profile modal is currently open. Used by edit / history / not-interested actions.
callingData
Array of devotees + their calling status for the current week. Loaded into Calling tab list.
attendanceCandidates
Map of id → devotee for the current attendance session. Used for fast PRESENT / UNDO updates without refetch.
devoteeSelectMode / selectedDevotees
Bulk-action multi-select state in the Devotees tab. Cleared on tab switch.

7 · Validation & Access Rules

🚫 Mark as "Not Interested"

Only superAdmin can call markNotInterested(). Hard-blocked in markNotInterested() function and the button is hidden via .super-admin-only class for all other roles. Devotee is then permanently excluded from all calling lists.

🚫 Calling By — must have login + same team

When saving a devotee profile, calling_by is validated against the users collection. The person must: (1) have a Firebase login and (2) be assigned to the same team as the devotee. Hard error on save if not found.

🚫 Facilitator — must have login (any team)

Same validation as Calling By, but without the team restriction. A facilitator (spiritual guide) can be from any team. Hard error on save if the name has no login in users collection.

🔒 Team Scoping (teamAdmin + serviceDevotee)

All Firestore queries for non-superAdmin users automatically append .where('teamName', '==', userTeam). The team dropdown is disabled in the UI via ft.disabled = true in applyRoleUI(). New devotee forms auto-fill the team field.

⚠ Calling Notes Required

When setting calling status to No / Maybe / Shift, calling notes are mandatory before saving. Only Yes status skips the notes requirement. Enforced in updateCallingStatus().

⚠ Late Submission Flag

Calling submissions after 9 PM are flagged as late. Visible in the Late Reports sub-tab under Calling. Based on updatedAtClient timestamp compared against the 21:00 hour cutoff.
